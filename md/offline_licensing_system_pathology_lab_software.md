# Licensing System Design
## Offline Electron + TypeScript Pathology Lab Software

---

## 1. Licensing Design Principles

1. Fully offline – software must function without internet
2. Vendor-controlled – only the vendor can generate valid licenses
3. Tamper-resistant – license modification must be detectable
4. Simple for labs – no technical activation steps
5. Scalable – supports resale, white-labeling, and add-ons

---

## 2. Licensing Model (Recommended)

### Offline Digitally Signed License File

The software operates based on a **digitally signed license file** generated by the vendor.

Characteristics:
- Generated only by vendor
- Installed locally by lab admin
- Cryptographically verified
- No online activation or phone-home logic

This model is ideal for labs with unreliable or no internet.

---

## 3. Supported License Types

| License Type | Purpose |
|-------------|---------|
| Trial | Pre-purchase evaluation |
| Annual | Subscription-based license |
| Perpetual | One-time purchase |
| Enterprise | All modules enabled |
| Add-on | Analyzer, Cloud Backup, QC |

---

## 4. What a License Controls

Licenses govern **features and usage**, not just expiry.

### Controlled Attributes
- Enabled modules (Billing, QC, Inventory, Analyzer, Backup)
- License duration (start / expiry)
- Maximum logical users
- Lab identity
- Machine binding

---

## 5. License File Structure

### File Name
```
license.lic
```

### Format: Signed JSON

```json
{
  "license_id": "LIC-2026-000142",
  "lab_name": "ABC Diagnostics",
  "issued_to": "ABC Diagnostics Pvt Ltd",
  "machine_id": "WIN-8F2A-93B1",
  "edition": "ENTERPRISE",
  "features": {
    "billing": true,
    "lab_report_entry": true,
    "qc": true,
    "inventory": true,
    "analyzer_integration": true,
    "cloud_backup": false
  },
  "max_users": 5,
  "issue_date": "2026-01-01",
  "expiry_date": "2027-01-01",
  "signature": "BASE64_ENCODED_SIGNATURE"
}
```

---

## 6. License Generation (Vendor Side)

1. Vendor maintains a private signing key
2. License JSON is generated with lab-specific data
3. License is cryptographically signed (RSA / ECC)
4. Signed license file is delivered to lab (email / USB)

The private key is **never** shipped with the software.

---

## 7. License Validation (Application Side)

### Validation Events
- Application startup
- Daily background check
- Access to restricted features (billing, report finalization)

### Validation Steps
1. Verify digital signature using embedded public key
2. Check expiry date
3. Validate machine binding (if enabled)
4. Load feature flags

### Validation States
- VALID
- EXPIRED
- INVALID
- TAMPERED

---

## 8. Machine Binding (Anti-Piracy)

### Machine Fingerprint Components (Windows)
- Windows Machine GUID
- Disk serial number
- CPU ID (hashed)

Fingerprint is stored as a SHA-256 hash.

### Binding Modes

| Mode | Usage |
|------|------|
| None | Trial |
| Soft | Annual license |
| Strict | Perpetual license |

Soft binding shows warnings; strict binding blocks execution on mismatch.

---

## 9. Trial License Behavior

- Time-limited (e.g., 15–30 days)
- Limited features (no analyzer, no cloud backup)
- Watermark on reports
- Cannot disable audit logging

Trial start date is stored securely to prevent clock rollback abuse.

---

## 10. License Renewal Flow

1. License nearing expiry → warning shown
2. Lab contacts vendor
3. Vendor issues new license file
4. Admin uploads new license
5. Application validates and continues

No internet required.

---

## 11. Admin License Management UI

Admin panel must show:
- License status
- Edition and enabled modules
- Expiry date
- Machine binding status

Actions:
- Upload new license file
- View license details

---

## 12. White-Label & Reseller Licensing

Two-tier model:

1. Vendor License – controls reseller rights
2. Lab License – controls lab usage

Enables multi-brand distribution with centralized control.

---

## 13. Grace Period & Fail-Safe Behavior

- Grace period: 7–14 days after expiry
- During grace: warnings only
- After grace:
  - Block new billing
  - Block report finalization
  - Allow viewing and exporting old reports

Prevents medico-legal risk.

---

## 14. Security Measures

| Threat | Mitigation |
|------|-----------|
| License tampering | Digital signature |
| Clock rollback | Last-run timestamp check |
| License sharing | Machine binding |
| Reverse engineering | Code obfuscation |

---

## 15. Implementation Stack (TypeScript)

- Crypto: Node.js `crypto`
- Hashing: SHA-256
- Signature verification: Public key verify
- Storage: Encrypted SQLite
- Packaging: electron-builder

---

## 16. Acceptance Criteria

- Invalid license never enables features
- Tampered license is detected
- Expired license blocks new operations
- Renewal does not affect existing data
- No internet dependency

---

## 17. Summary

This offline licensing system:
- Aligns with offline-first architecture
- Is secure and audit-friendly
- Works for small labs and resellers
- Minimizes support and operational risk

This design is suitable for long-term commercial deployment.
